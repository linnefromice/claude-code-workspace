# /learn-eval - 抽出、評価、保存

`/learn` を品質ゲート、保存場所の決定、知識配置の認識で拡張し、スキルファイルを書き込む前に検証します。

## 抽出対象

以下を探します:

1. **エラー解決パターン** — 根本原因 + 修正 + 再利用可能性
2. **デバッグテクニック** — 自明でないステップ、ツールの組み合わせ
3. **ワークアラウンド** — ライブラリの癖、API制限、バージョン固有の修正
4. **プロジェクト固有のパターン** — 規約、アーキテクチャの判断、統合パターン

## プロセス

1. 抽出可能なパターンについてセッションをレビュー
2. 最も価値のある/再利用可能なインサイトを特定

3. **保存場所の決定:**
   - 質問: 「このパターンは別のプロジェクトで役立つか？」
   - **グローバル** (`~/.claude/skills/learned/`): 2つ以上のプロジェクトで使用可能な汎用パターン（bash互換性、LLM API動作、デバッグテクニックなど）
   - **プロジェクト** (現在のプロジェクトの `.claude/skills/learned/`): プロジェクト固有の知識（特定の設定ファイルの癖、プロジェクト固有のアーキテクチャ判断など）
   - 迷った場合はグローバルを選択（グローバル → プロジェクトへの移動は逆より容易）

4. 以下のフォーマットでスキルファイルをドラフト:

```markdown
---
name: pattern-name
description: "130文字以内"
user-invocable: false
origin: auto-extracted
---

# [説明的なパターン名]

**抽出日:** [日付]
**コンテキスト:** [これが適用される場面の簡潔な説明]

## 問題
[このパターンが解決する問題 - 具体的に]

## 解決策
[パターン/テクニック/ワークアラウンド - コード例付き]

## 使用タイミング
[トリガー条件]
```

5. **品質ゲート — チェックリスト + 総合判定**

   ### 5a. 必須チェックリスト（実際にファイルを読んで検証）

   ドラフトを評価する前に以下を**すべて**実行:

   - [ ] `~/.claude/skills/` と関連プロジェクトの `.claude/skills/` ファイルをキーワードでGrepしてコンテンツの重複を確認
   - [ ] MEMORY.md（プロジェクトとグローバル両方）の重複を確認
   - [ ] 既存スキルへの追記で十分かどうかを検討
   - [ ] これが再利用可能なパターンであり、一回限りの修正ではないことを確認

   ### 5b. 総合判定

   チェックリスト結果とドラフト品質を総合し、以下の**いずれか**を選択:

   | 判定 | 意味 | 次のアクション |
   |------|------|--------------|
   | **Save** | ユニーク、具体的、スコープが適切 | ステップ6に進む |
   | **Improve then Save** | 価値があるが改善が必要 | 改善点をリスト → 修正 → 再評価（1回） |
   | **Absorb into [X]** | 既存スキルに追記すべき | ターゲットスキルと追加内容を表示 → ステップ6 |
   | **Drop** | 些細、冗長、または抽象的すぎる | 理由を説明して停止 |

**ガイドラインディメンション**（判定の参考、スコアリングはしない）:

- **具体性と実行可能性**: すぐに使用可能なコード例やコマンドを含む
- **スコープの適合性**: 名前、トリガー条件、内容が整合し、単一パターンに焦点を当てている
- **ユニーク性**: 既存スキルでカバーされていない価値を提供（チェックリスト結果に基づく）
- **再利用可能性**: 将来のセッションで現実的なトリガーシナリオが存在する

6. **判定別の確認フロー**

- **Improve then Save**: 必要な改善 + 修正ドラフト + 更新されたチェックリスト/判定を1回の再評価後に提示。修正後の判定が **Save** なら、ユーザー確認後に保存。それ以外は新しい判定に従う
- **Save**: 保存パス + チェックリスト結果 + 1行の判定理由 + 完全なドラフトを提示 → ユーザー確認後に保存
- **Absorb into [X]**: ターゲットパス + 追加（diff形式）+ チェックリスト結果 + 判定理由を提示 → ユーザー確認後に追記
- **Drop**: チェックリスト結果 + 理由のみを表示（確認不要）

7. 決定された場所に保存/追記

## ステップ5の出力フォーマット

```
### チェックリスト
- [x] skills/ grep: 重複なし（または: 重複発見 → 詳細）
- [x] MEMORY.md: 重複なし（または: 重複発見 → 詳細）
- [x] 既存スキルへの追記: 新規ファイルが適切（または: [X]に追記すべき）
- [x] 再利用可能性: 確認済み（または: 一回限り → Drop）

### 判定: Save / Improve then Save / Absorb into [X] / Drop

**理由:** （判定を説明する1-2文）
```

## 注意事項

- 些細な修正を抽出しない（タイポ、単純な構文エラー）
- 一回限りの問題を抽出しない（特定のAPI障害など）
- 将来のセッションで時間を節約するパターンに焦点
- スキルは焦点を絞る — 1パターンにつき1スキル
- 判定がAbsorbの場合、新しいファイルを作成するのではなく既存スキルに追記
