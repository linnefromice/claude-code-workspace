# Web パフォーマンスルール

## Core Web Vitals の目標値

| 指標 | 目標値 |
|--------|--------|
| LCP | < 2.5s |
| INP | < 200ms |
| CLS | < 0.1 |
| FCP | < 1.5s |
| TBT | < 200ms |

## バンドルバジェット

| ページタイプ | JS バジェット（gzip 後） | CSS バジェット |
|-----------|---------------------|------------|
| ランディングページ | < 150kb | < 30kb |
| アプリページ | < 300kb | < 50kb |
| マイクロサイト | < 80kb | < 15kb |

## ロード戦略

1. 正当な場合は above-the-fold のクリティカル CSS をインライン化する
2. プリロードはヒーロー画像とプライマリフォントのみに限定する
3. 非クリティカルな CSS / JS は defer する
4. 重いライブラリは動的インポートする

```js
const gsapModule = await import('gsap');
const { ScrollTrigger } = await import('gsap/ScrollTrigger');
```

## 画像の最適化

- 明示的な `width` と `height`
- `loading="eager"` と `fetchpriority="high"` はヒーローメディアにのみ使用
- below-the-fold のアセットには `loading="lazy"`
- AVIF または WebP をフォールバック付きで優先
- レンダリングサイズを大きく超えるソース画像を配信しない

## フォントのロード

- 明確な例外がなければフォントファミリーは最大 2 つまで
- `font-display: swap`
- 可能な限りサブセット化する
- 本当にクリティカルな weight / style のみプリロードする

## アニメーションのパフォーマンス

- コンポジタフレンドリーなプロパティのみをアニメートする
- `will-change` は狭い範囲で使い、終わったら外す
- シンプルなトランジションには CSS を優先する
- JS アニメーションには `requestAnimationFrame` か確立されたアニメーションライブラリを使う
- スクロールハンドラの消耗を避け、IntersectionObserver や行儀の良いライブラリを使う

## パフォーマンスチェックリスト

- [ ] すべての画像に明示的な寸法がある
- [ ] 意図しないレンダリングブロックリソースがない
- [ ] 動的コンテンツによるレイアウトシフトがない
- [ ] モーションがコンポジタフレンドリーなプロパティに留まっている
- [ ] サードパーティスクリプトが async/defer でロードされ、必要なときだけ読み込まれている
