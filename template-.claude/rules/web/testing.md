# Web テストルール

## 優先順位

### 1. ビジュアルリグレッション

- 主要なブレークポイントでスクリーンショットを取る: 320、768、1024、1440
- ヒーローセクション、scrollytelling セクション、意味のある状態をテストする
- ビジュアル重視の作業には Playwright のスクリーンショットを使う
- 両テーマがある場合は両方テストする

### 2. アクセシビリティ

- 自動アクセシビリティチェックを実行する
- キーボードナビゲーションをテストする
- reduced-motion の挙動を検証する
- カラーコントラストを検証する

### 3. パフォーマンス

- 意味のあるページに対して Lighthouse または同等のツールを実行する
- [performance.md](performance.md) の CWV 目標を維持する

### 4. クロスブラウザ

- 最低: Chrome、Firefox、Safari
- スクロール、モーション、フォールバック挙動をテストする

### 5. レスポンシブ

- 320、375、768、1024、1440、1920 でテストする
- オーバーフローがないことを検証する
- タッチ操作を検証する

## E2E の形

```ts
import { test, expect } from '@playwright/test';

test('landing hero loads', async ({ page }) => {
  await page.goto('/');
  await expect(page.locator('h1')).toBeVisible();
});
```

- タイムアウトに依存する flaky なアサーションを避ける
- 決定論的な待機を優先する

## ユニットテスト

- ユーティリティ、データ変換、カスタムフックをテストする
- 非常にビジュアル寄りのコンポーネントでは、脆いマークアップアサーションより、ビジュアルリグレッションの方がシグナルとして価値が高いことが多い
- ビジュアルリグレッションはカバレッジ目標を補完するものであり、置き換えるものではない
