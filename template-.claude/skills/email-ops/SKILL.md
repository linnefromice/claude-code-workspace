# Email Ops

実際のタスクがメールボックス作業である場合に使用します: トリアージ、ドラフティング、返信、送信、またはメッセージが Sent に到着したことの証明。

これは汎用ライティングスキルではありません。実際のメールサーフェス周辺のオペレーターワークフローです。

## Skill Stack

関連する場合、以下の ECC ネイティブスキルをワークフローに取り込みます:

- ユーザー向けの何かをドラフトする前に `brand-voice`
- 投資家、パートナー、またはスポンサー向けメールには `investor-outreach`
- スレッドが汎用の通信ではなく課金/サポートインシデントである場合は `customer-billing-ops`
- メッセージまたはスレッドを後で永続的なコンテキストに取り込む必要がある場合は `knowledge-ops`
- 返信が最新の外部事実に依存する場合は `research-ops`

## 使用タイミング

- ユーザーが inbox のトリアージまたは低シグナルメールのアーカイブを要求する
- ユーザーがドラフト、返信、または新規送信メールを望む
- ユーザーがメールが既に送信されたかどうかを知りたい
- ユーザーがどのアカウント、スレッド、または Sent エントリが使用されたかの証明を望む

## ガードレール

- ユーザーがライブ送信を明確に要求しない限り、まずドラフトする
- 本物の Sent フォルダまたはクライアント側の確認なしにメッセージが送信されたと主張しない
- 送信者アカウントをカジュアルに切り替えない。プロジェクトと受信者にマッチするアカウントを選ぶ
- クリーンアップ中に不確実なビジネスメールを削除しない
- タスクが実際に DM または iMessage 作業である場合、`messages-ops` に引き継ぐ

## ワークフロー

### 1. 正確なサーフェスを解決する

アクション前に以下を確定:

- どのメールボックスアカウント
- どのスレッドまたは受信者
- タスクがトリアージ、ドラフト、返信、または送信か
- ユーザーがドラフトのみかライブ送信を望むか

### 2. 作成前にスレッドを読む

返信の場合:

- 既存のスレッドを読む
- 最後の outbound タッチを特定
- コミットメント、期限、未回答の質問を特定

新規送信を作成する場合:

- warmth レベルを特定
- 正しいチャネルと送信者アカウントを選択
- ドラフト前に `brand-voice` を取り込む

### 3. ドラフトしてから検証する

ドラフトのみの作業の場合:

- 最終コピーを生成
- 送信者、受信者、件名、目的を述べる

ライブ送信作業の場合:

- 最終本文を先に検証
- 選択したメールサーフェスを通して送信
- メッセージが Sent または同等の送信済みコピーストアに到着したことを確認

### 4. 正確な状態を報告する

正確なステータス語を使用:

- drafted
- approval-pending
- sent
- blocked
- awaiting verification

送信サーフェスがブロックされている場合、ドラフトを保存し、そう言わずに第二のトランスポートを即興せず、正確なブロッカーを報告します。

## 出力フォーマット

```text
MAIL SURFACE
- account
- thread / recipient
- requested action

DRAFT
- subject
- body

STATUS
- drafted / sent / blocked
- proof of Sent when applicable

NEXT STEP
- send
- follow up
- archive / move
```

## 落とし穴

- 送信済みコピーチェックなしで送信成功を主張しない
- スレッド履歴を無視してコンテキストなしの返信を書かない
- メールボックス作業を DM またはテキストメッセージのワークフローと混在させない
- シークレット、認証詳細、または不要なメッセージメタデータを露出しない

## 検証

- 応答がアカウントとスレッドまたは受信者を名指しする
- 送信主張には Sent の証明または明示的なクライアント側確認が含まれる
- 最終状態が drafted / sent / blocked / awaiting verification のいずれか
