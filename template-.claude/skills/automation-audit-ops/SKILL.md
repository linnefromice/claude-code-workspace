# Automation Audit Ops

どの自動化がライブか、どのジョブが壊れているか、どこに重複があるか、どのツールやコネクタが今実際に有用な仕事をしているかをユーザーが尋ねた場合に使用します。

これは監査優先のオペレータースキルです。仕事は、何かを書き直す前に、エビデンスに裏付けられたインベントリと keep / merge / cut / fix-next の推奨セットを生成することです。

## Skill Stack

関連する場合、以下の ECC ネイティブスキルをワークフローに取り込みます:

- コネクタ、MCP、フック、アプリのインベントリには `workspace-surface-audit`
- 監査がライブリポジトリの真実を永続的なコンテキストと整合させる必要がある場合は `knowledge-ops`
- 答えが CI、スケジュールされたワークフロー、issue、または PR 自動化に依存する場合は `github-ops`
- 実際の問題が兄弟アプリリポジトリでの webhook ファンアウト、キューイングされたジョブ、または課金消費である場合は `ecc-tools-cost-audit`
- ローカルインベントリを現在のプラットフォームサポートまたは公開ドキュメントと比較する必要がある場合は `research-ops`
- 想定されるリカバリに頼るのではなく、修正後の状態を証明するには `verification-loop`

## 使用タイミング

- ユーザーが「どんな自動化があるか」「何がライブか」「何が壊れているか」「何が重複しているか」と尋ねる
- タスクが cron ジョブ、GitHub Actions、ローカルフック、MCP サーバー、コネクタ、ラッパー、またはアプリ統合にまたがる
- ユーザーが別のエージェントシステムから何が移植されたか、ECC 内で何をまだ再構築する必要があるかを知りたい
- ワークスペースに同じことを行う複数の方法が蓄積され、ユーザーが 1 つの正準レーンを望んでいる

## ガードレール

- ユーザーが修正を明示的に要求しない限り、read-only で開始する
- 以下を区別する:
  - 設定済み
  - 認証済み
  - 最近検証済み
  - 古いまたは壊れている
  - 完全に欠落している
- スキルや設定が参照しているだけで、ツールがライブだと主張しない
- エビデンステーブルが存在するまで、重複するサーフェスをマージまたは削除しない

## ワークフロー

### 1. 実際のサーフェスをインベントリする

理論化する前に現在のライブサーフェスを読みます:

- リポジトリフックとローカルフックスクリプト
- GitHub Actions とスケジュールされたワークフロー
- MCP 設定と有効なサーバー
- コネクタまたはアプリベースの統合
- ラッパースクリプトとリポジトリ固有の自動化エントリーポイント

サーフェスごとにグループ化します:

- ローカルランタイム
- リポジトリ CI / 自動化
- 接続された外部システム
- メッセージング / 通知
- 課金 / カスタマーオペレーション
- リサーチ / モニタリング

### 2. 各アイテムをライブ状態で分類する

浮上したすべての自動化について以下をマーク:

- 設定済み
- 認証済み
- 最近検証済み
- 古いまたは壊れている
- 欠落

次に問題タイプを分類:

- アクティブな破損
- 認証停止
- 古いステータス
- 重複または冗長性
- 欠落した機能

### 3. 証明パスをトレースする

重要な主張すべてを具体的なソースで裏付けます:

- ファイルパス
- ワークフロー実行
- フックログ
- 設定エントリ
- 最近のコマンド出力
- 正確な失敗シグネチャ

現在の状態が曖昧であれば、監査が完了したふりをせず、直接そう述べます。

### 4. keep / merge / cut / fix-next で締めくくる

重複する、または疑わしいサーフェスそれぞれについて、1 つのコールを返します:

- keep
- merge
- cut
- fix next

価値は、ノイジーな自動化を 1 つの正準 ECC レーンに集約することにあり、すべての歴史的パスを保存することにはありません。

## 出力フォーマット

```text
CURRENT SURFACE
- automation
- source
- live state
- proof

FINDINGS
- active breakage
- overlap
- stale status
- missing capability

RECOMMENDATION
- keep
- merge
- cut
- fix next

NEXT ECC MOVE
- exact skill / hook / workflow / app lane to strengthen
```

## 落とし穴

- ライブインベントリが読める場合に記憶から答えない
- 「設定内に存在」を「動作中」として扱わない
- 壊れている高シグナルパスを名指しする前に低価値の冗長性を修正しない
- ユーザーが最初にインベントリを求めた場合、タスクをリポジトリ書き直しに拡大しない

## 検証

- 重要な主張はライブの証明パスを引用する
- 浮上した自動化それぞれは明確なライブ状態カテゴリでラベル付けされる
- 最終推奨が keep / merge / cut / fix-next を区別する
