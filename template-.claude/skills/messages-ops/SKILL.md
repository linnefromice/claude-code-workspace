# Messages Ops

タスクがライブメッセージ取得（iMessage、DM、最近のワンタイムコード、フォローアップ前のスレッド確認）である場合にこれを使用します。

これはメール作業ではありません。主なサーフェスがメールボックスである場合は `email-ops` を使用してください。

## スキルスタック

関連する場合、以下の ECC ネイティブスキルをワークフローに取り込みます:

- メッセージタスクが実際にはメールボックス作業である場合は `email-ops`
- DM スレッドがアウトバウンドネットワーク作業に属する場合は `connections-optimizer`
- ライブスレッドがターゲティングやウォームパスのアウトリーチに情報を与えるべき場合は `lead-intelligence`
- スレッド内容を永続的なコンテキストに捕捉する必要がある場合は `knowledge-ops`

## 使用タイミング

- ユーザーが "メッセージを読んで"、"テキストを確認"、"DM を見て"、"コードを見つけて" と言った場合
- タスクがライブスレッドや、ローカルメッセージングサーフェスに配信された最近のコードに依存する場合
- ユーザーがどのソースまたはスレッドが検査されたかの証明を望む場合

## ガードレール

- まずソースを解決する:
  - ローカルメッセージ
  - X / ソーシャル DM
  - ブラウザゲート付きの他のメッセージサーフェス
- ソースを明示せずにスレッドを確認したと主張しない
- チェック済みのヘルパーや標準パスが存在する場合、生のデータベースアクセスを即興で行わない
- 認証や MFA がサーフェスをブロックしている場合、正確なブロッカーを報告する

## ワークフロー

### 1. 正確なスレッドを解決する

他の何かをする前に、次を確定する:

- メッセージサーフェス
- 送信者 / 受信者 / サービス
- 時間ウィンドウ
- タスクが取得、検査、または返信準備のいずれか

### 2. 下書き前に読む

タスクがアウトバウンドのフォローアップに変わる可能性がある場合:

- 最新のインバウンドを読む
- オープンループを特定する
- その後、必要に応じて適切なアウトバウンドスキルにハンドオフする

### 3. コードをフォーカスされた取得タスクとして処理する

ワンタイムコードの場合:

- まず最近のローカルメッセージウィンドウを検索
- 可能な場合はサービスまたは送信者で絞り込む
- コードが見つかるか、フォーカスされた検索が尽きたら停止する

### 4. 正確なエビデンスを報告する

以下を返す:

- 使用したソース
- 可能な場合はスレッドまたは送信者
- 時間ウィンドウ
- 正確なステータス:
  - read
  - code-found
  - blocked
  - awaiting reply draft

## 出力フォーマット

```text
SOURCE
- メッセージサーフェス
- 送信者 / スレッド / サービス

RESULT
- メッセージサマリーまたはコード
- 時間ウィンドウ

STATUS
- read / code-found / blocked / awaiting reply draft
```

## 落とし穴

- メールボックス作業と DM/テキスト作業を混同しない
- ソースを明示せずに取得を主張しない
- 依頼が最近のコード検索である場合、広範な検索に時間を使わない
- ブロックされた認証パスをブロッカーを表面化せずにリトライし続けない

## 検証

- 応答がメッセージソースを明示している
- 応答に送信者、サービス、スレッド、または明確なブロッカーが含まれている
- 最終状態が明示的で境界が明確である
